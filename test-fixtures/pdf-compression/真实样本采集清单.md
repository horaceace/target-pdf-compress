# PDF 压缩真实样本采集清单

更新日期：2026-05-31

## 目的

当前合成样本已经能验证 benchmark 流程，但还不能代表真实用户文件。

这里的“真实样本”不是要求上传隐私原件，而是要求使用非敏感、脱敏或公开来源的 PDF，尽量模拟真实用户会上传的文件类型。当前状态里的 `0/8` 表示 8 个必备真实场景槽位还没有放入可测试 PDF。

下一步强压方案选择必须基于真实样本：

- 当前浏览器压缩
- 浏览器 `Scanned PDF` render-and-rebuild
- qpdf / qpdf-wasm
- pdfcpu
- Ghostscript 效果对比

这里的“外部引擎”是指复用成熟开源 PDF 工具，而不是自己从 0 写底层压缩算法：

- `qpdf`：优先验证，主要做 PDF 结构优化、线性化、对象流压缩
- `pdfcpu`：服务端优化候选
- Ghostscript：强压扫描件和图片型 PDF 的效果对比候选，生产前必须确认 license、部署和画质风险

外部引擎状态里的 `0/3` 表示当前机器没有检测到 `qpdf / pdfcpu / gs` 这 3 个命令。

## 样本原则

- 只使用非敏感文件
- 不放身份证、合同原件、真实银行流水、真实病历等隐私文件
- 如果使用真实文件，先脱敏
- 导入前先跑隐私预检，检查标题、作者、关键词、创建工具、文件名等基础风险
- 文件放在 `test-fixtures/pdf-compression/`
- 文件名用英文小写和短横线

## 必备样本

| 文件名建议 | 类型 | 目标 | 验收重点 |
|---|---|---|---|
| `real-resume-export.pdf` | 简历 / 求职材料 | 测试 job application 场景 | 清晰度、是否仍适合上传 |
| `real-form-upload.pdf` | 表单 / 申请材料 | 测试 portal upload 场景 | 是否能压到常见限制附近 |
| `real-scanned-color.pdf` | 彩色扫描件 | 测试扫描件强压 | 文字可读、章/表格不糊 |
| `real-scanned-bw.pdf` | 黑白扫描件 | 测试低色彩扫描 | 文字边缘、文件体积 |
| `real-image-heavy.pdf` | 图片型 PDF / 截图 PDF | 测试图片重建路径 | 图片细节和压缩率平衡 |
| `real-invoice-or-table.pdf` | 发票 / 表格类 PDF | 测试线条和数字 | 表格线、数字可读性 |
| `real-large-upload.pdf` | 大文件 PDF | 测试性能和失败路径 | 耗时、内存、是否卡死 |
| `real-email-attachment.pdf` | 邮件附件类 PDF | 测试一般分享场景 | 体积是否明显下降 |

## 记录字段

每个样本需要记录：

- 原始大小
- 页数
- 文件类型
- 当前最佳模式
- 浏览器 scanned 模式结果
- 是否达到目标大小
- 肉眼可读性
- 是否适合作为回归样本

## 下一步命令

查看还缺哪些真实样本：

```bash
npm run samples:import -- --list
```

生成一个可交付给采样人的 intake 目录：

```bash
npm run samples:intake-kit
```

默认输出：

```text
test-fixtures/pdf-compression/results/real-sample-intake-kit/
```

目录里会生成：

- `HANDOFF.md`：给采样人的一页式操作说明
- `README.md`：导入流程和隐私规则
- `CHECKLIST.md`：每个样本槽位的人工检查项
- `SAMPLE-SLOTS.csv`：可复制到表格工具里的槽位清单

导入前做隐私预检：

```bash
npm run samples:privacy-check -- --source /path/to/non-sensitive.pdf --write-report
```

如果预检提示 PDF 元数据风险，先生成一个清理元数据后的副本：

```bash
npm run samples:sanitize -- --source /path/to/source.pdf --output /path/to/sanitized.pdf
```

然后对清理后的文件再跑一次预检：

```bash
npm run samples:privacy-check -- --source /path/to/sanitized.pdf --write-report
```

导入一个本地非敏感 PDF 到指定样本槽位：

```bash
npm run samples:import -- --slot real-scanned-color.pdf --source /path/to/sanitized.pdf --note "sanitized color scan"
```

如果已经把一批 PDF 按 manifest 槽位命名好，例如 `real-resume-export.pdf`、`real-scanned-color.pdf`，可以先 dry-run 批量预览：

```bash
npm run samples:import -- --source-dir /path/to/renamed-pdfs --dry-run
```

确认无误后批量导入：

```bash
npm run samples:import -- --source-dir /path/to/renamed-pdfs
```

如果预检提示元数据或文件名风险，先脱敏再导入。只有人工确认风险可接受时，才允许：

```bash
npm run samples:import -- --slot real-scanned-color.pdf --source /path/to/non-sensitive.pdf --accept-privacy-risk
```

检查真实样本覆盖：

```bash
npm run samples:check
```

生成合成样本：

```bash
npm run fixtures:compression
```

跑当前 Node baseline：

```bash
npm run benchmark:compression
```

跑外部引擎可用性和效果对比：

```bash
npm run benchmark:external-setup
npm run benchmark:external-compressors
```

如果 qpdf / pdfcpu / Ghostscript 不在 `PATH`，可以指定本地二进制：

```bash
QPDF_BIN=/absolute/path/to/qpdf npm run benchmark:external-compressors
PDFCPU_BIN=/absolute/path/to/pdfcpu npm run benchmark:external-compressors
GS_BIN=/absolute/path/to/gs npm run benchmark:external-compressors
```

跑浏览器 scanned 路径：

```bash
npm run dev
```

打开：

```text
http://localhost:3000/dev/compression-benchmark
```

执行：

- `Run all local fixtures`
- 优先点击 `Save suite files`，直接写入 `test-fixtures/pdf-compression/results/`
- 如需手动备份，再点击 `Export suite Markdown`

导出的 suite Markdown 应包含：

- `Fixture mix`
- `Type` 列
- `real` / `synthetic` 样本区分

只有 `real-*` 样本结果能作为强压引擎接入决策的核心证据；`sample-*` 合成样本只用于流程回归和方向判断。

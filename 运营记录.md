### 今日基准结论

当前 Node benchmark 只测量共享的 `pdf-lib` 压缩 pass，不包含浏览器专用的 scanned render path。

结果显示：

- `sample-clean-office.pdf`：最佳模式 `Balanced`，仅缩小 `0.3%`
- `sample-mixed-content.pdf`：最佳模式 `Balanced`，仅缩小 `0.6%`
- `sample-image-heavy.pdf`：最佳模式 `Balanced`，约 `0%`
- `sample-scanned-bw.pdf`：最佳模式 `Balanced`，约 `0%`
- `sample-scanned-color.pdf`：最佳模式 `Balanced`，约 `0%`
- `sample-scanned-like.pdf`：最佳模式 `Balanced`，约 `0%`

判断：

- 继续微调 `pdf-lib` 保存参数不会解决图片型和扫描件压缩问题
- 下一步应验证成熟压缩引擎，而不是从 0 写压缩算法
- 第一优先级是 `qpdf-wasm / qpdf`
- 第二优先级是 `pdfcpu`
- Ghostscript 只做效果对比，license 和部署确认前不上生产

### 浏览器 scanned 路径实测

已通过开发页运行完整浏览器 benchmark：

- 页面：`http://localhost:3000/dev/compression-benchmark`
- 操作：`Run all local fixtures`
- 导出结果：
  - `test-fixtures/pdf-compression/results/browser-compression-benchmark-suite-2026-05-31.md`

关键结果：

- `sample-image-heavy.pdf`：`Scanned PDF` 最佳，`8.5 MB -> 265 KB`，缩小 `97%`
- `sample-scanned-bw.pdf`：`Scanned PDF` 最佳，`614 KB -> 203 KB`，缩小 `67%`
- `sample-scanned-color.pdf`：`Scanned PDF` 最佳，`7.8 MB -> 190 KB`，缩小 `98%`
- `sample-scanned-like.pdf`：`Scanned PDF` 最佳，`214 KB -> 92 KB`，缩小 `57%`
- `sample-clean-office.pdf` 和 `sample-mixed-content.pdf` 只缩小约 `1%`

判断：

- 当前浏览器端 `Scanned PDF` render-and-rebuild 路径对图片型 / 扫描型 PDF 有明显效果
- Node benchmark 中的 `0%` 主要是因为它没有覆盖浏览器 canvas 重建路径
- 下一步不应简单判定“现有压缩无效”，而应区分：
  - 普通 office PDF：本来已经很小，压缩收益低
  - 图片型 / 扫描型 PDF：现有 `Scanned PDF` 模式有效，但需要用真实样本继续验证画质
  - 严格目标大小 / 超大文件：仍需 qpdf / pdfcpu / Ghostscript 或服务端强压对比

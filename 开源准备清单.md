# 开源准备清单

## 当前结论

当前仓库已经具备开源前的大部分工程材料，但还不能直接公开为开源项目。

原因不是代码不能看，而是开源元信息还没有完成最终决策：

- 还没有提交 `LICENSE` 或 `LICENSE.md`
- `package.json` 还没有声明 `license`
- `package.json` 仍是 `"private": true`

在这三个决策完成前，仓库可以继续作为私有产品仓库推进，但不建议公开宣传为开源项目。

## 推荐开源定位

项目开源时，建议定位为：

> FileSmaller: browser-first PDF compression and upload-limit PDF tools.

不要定位为：

- 完整 PDF 平台
- Stirling-PDF 替代品
- 服务端强压缩系统
- qpdf / pdfcpu / Ghostscript 生产封装

当前最适合公开的价值是：轻量、浏览器优先、出海 SEO 工具站、上传大小限制场景、真实压缩效果验证流程。

## License 决策

需要人工确认后再改文件。

推荐选项：

- `MIT`：最简单宽松，适合工具站、模板、独立开发者项目，推荐默认选这个。
- `Apache-2.0`：同样宽松，但包含更明确的专利授权条款，适合希望降低专利争议风险的项目。
- 暂不开源：继续保持私有仓库，只公开产品网站。

不建议在没有明确商业策略前选择强 copyleft 许可证，例如 `GPL` / `AGPL`，因为后续商业化和集成会更受限制。

## package.json 决策

正式开源前需要同步修改：

```json
{
  "private": false,
  "license": "MIT"
}
```

如果选择 `Apache-2.0`，则写：

```json
{
  "private": false,
  "license": "Apache-2.0"
}
```

未确认 License 前不要改成 `private: false`。

## 不能提交到公开仓库的内容

继续保持以下内容不进入 git：

- 真实用户 PDF
- 简历、发票、合同、申请表、扫描件、账单等真实样本
- 从真实样本生成的 benchmark results
- 包含本机路径、客户信息、个人信息的报告

当前 `.gitignore` 策略应继续保留：

```text
test-fixtures/pdf-compression/*.pdf
test-fixtures/pdf-compression/results/*
```

## 可以公开的内容

可以公开：

- Next.js / Cloudflare Workers 产品代码
- SEO 页面配置和工具页面模板
- 浏览器端 PDF 压缩、合并、拆分、旋转、页面删除/重排、PDF/JPG 转换代码
- synthetic fixture 生成脚本
- 真实样本采集流程文档
- qpdf / pdfcpu / Ghostscript benchmark 脚本
- 发布检查、SEO 检查、CI 检查、postdeploy smoke check
- 产品规划和开源复用调研，但需避免泄露私人运营数据

## 强压缩边界

开源不等于强压缩已完成。

当前仍不能公开宣传：

- qpdf 已生产上线
- pdfcpu 已生产上线
- Ghostscript 已生产上线
- 服务端强压缩已验证
- 保证压到指定大小

强压缩能力仍需通过：

```bash
npm run benchmark:decision-gate
npm run benchmark:engine-recommendation
```

## 开源前检查命令

```bash
npm run release:open-source-check
npm run release:ci-check
npm run release:standard-check
```

当前 `release:open-source-check` 允许作为开源元信息门禁。

通过前不要公开仓库；通过后仍需人工扫一遍 README、规划文档、运营记录，确认没有私有信息。

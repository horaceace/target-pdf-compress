# Split PDF 技术方案

## 目标

在现有 `Compress PDF` 和 `Merge PDF` 的基础上，继续推进产品规划里的下一个矩阵工具：

- `Split PDF`

目标不是做复杂的 PDF 编辑器，而是先做一个简单、清晰、可上线的 PDF 拆分工具。

## 一、产品目标

用户可以：

1. 上传一个 PDF
2. 选择拆分方式
3. 生成多个拆分后的 PDF
4. 下载拆分结果

第一版优先满足最常见需求：

- 按页范围拆分
- 提取单独页
- 导出多个拆分文件

第一版不做：

- 登录
- 云端保存
- 历史记录
- 页面缩略图拖拽
- 页面级可视化编辑器
- 服务端大文件任务队列

## 二、为什么现在做 Split PDF

根据产品规划，`Split PDF` 是 `Merge PDF` 之后最顺手的矩阵工具，原因很明确：

1. 和压缩用户高度重合
- 很多用户先拆分，再压缩
- 也有用户先拆分，再上传指定部分

2. 和 Merge PDF 自然形成一组
- 一个是“合并多个 PDF”
- 一个是“拆开一个 PDF”

3. 可以继续复用当前技术栈
- `Next.js`
- `TypeScript`
- `pdf-lib`
- 浏览器内处理

4. 对首页导航、内链和后续 SEO 都友好

## 三、第一版功能范围

### 第一版必须做

- 单个 PDF 上传
- 显示总页数
- 按页范围拆分
- 支持输入多个范围
- 为每个范围生成一个新 PDF
- 支持逐个下载
- 支持全部下载
- 错误提示和输入校验

### 第一版推荐做法

第一版不建议一上来做太多拆分模式，建议只做这一种核心模式：

- `Split by page ranges`

例如：

- `1-2`
- `3-5`
- `6`

用户输入多个范围后，系统生成多个 PDF：

- `file-part-1-2.pdf`
- `file-part-3-5.pdf`
- `file-part-6.pdf`

### 第二版再考虑

- 每页单独拆分
- 按固定页数拆分
- 提取首页 / 最后一页
- 页面缩略图预览
- 拖拽式页级选择
- Zip 打包下载

## 四、技术方案

### 技术基线

当前项目已经具备：

- `Next.js`
- `TypeScript`
- `pdf-lib`

所以 `Split PDF` 第一版继续沿用浏览器内处理，不引入后端服务。

### 拆分方式

使用 `pdf-lib`：

1. 读取上传的 PDF 文件
2. 获取总页数
3. 解析用户输入的页范围
4. 针对每个范围创建一个新的 `PDFDocument`
5. 使用 `copyPages`
6. 将目标页加入新文档
7. 导出多个拆分后的 PDF

### 为什么选页范围模式

因为它覆盖面最大，而且第一版实现成本最低。

典型使用场景：

- 从完整合同中拆出签字页
- 从多页材料中拆出某几页上传
- 从扫描件中拆出证件页
- 从大型 PDF 中拆出需要压缩的局部内容

## 五、建议文件结构

建议新增：

- `lib/pdf/split.ts`
- `components/split-pdf-card.tsx`
- `app/split-pdf/page.tsx`

后续如果做更复杂的页范围解析，也可以新增：

- `lib/pdf/split-range.ts`

## 六、UI 结构

建议页面结构：

1. 标题区
- `Split PDF files`
- 简短说明

2. 上传区
- 拖拽 / 选择一个 PDF

3. 文件信息区
- 文件名
- 文件大小
- 总页数

4. 拆分设置区
- 页范围输入框
- 示例说明

5. 主操作区
- `Split PDF`
- 清空 / 重新上传

6. 结果区
- 每个拆分结果一张卡片
- 范围信息
- 文件名
- 下载按钮
- 全部下载按钮

## 七、页范围输入规则

第一版建议支持：

- 单页：`6`
- 范围：`3-5`
- 多段：`1-2,4,7-9`

输入规则：

1. 页码从 `1` 开始
2. 起始页不能大于结束页
3. 不能超过总页数
4. 允许空格，但内部会统一清洗
5. 非法输入要给出明确错误

错误示例：

- `0-2`
- `5-3`
- `abc`
- `1-999` 当总页数不足时

提示原则：

- 不只告诉用户“错了”
- 要明确告诉他哪里错、怎么改

例如：

- `Page range 8-12 exceeds the total page count of this PDF.`
- `Use formats like 1-3, 5, 7-9.`

## 八、状态设计

建议状态：

- `selectedFile`
- `pageCount`
- `rangeInput`
- `parsedRanges`
- `results[]`
- `error`
- `isProcessing`

其中 `results[]` 每项建议包含：

- `id`
- `label`
- `fileName`
- `blob`
- `pageStart`
- `pageEnd`
- `pageTotal`

## 九、核心处理流程

### 上传后

1. 校验文件类型
2. 校验文件大小
3. 读取 PDF
4. 获取总页数
5. 显示文件元信息

### 拆分时

1. 解析范围输入
2. 校验范围合法性
3. 为每个范围创建新 `PDFDocument`
4. 复制对应页
5. 导出结果数组
6. 渲染下载区

## 十、风险

### 1. 大 PDF 的浏览器内存压力

第一版接受这个限制，不做服务端拆分。

### 2. 范围输入容易出错

所以第一版必须把输入规则讲清楚，并提供错误提示。

### 3. 多个结果下载体验一般

第一版先接受多个单独下载按钮。

如果后续用户确实需要，再补：

- Zip 打包下载

### 4. 加密或损坏 PDF

要给出明确错误提示，不要只抛技术异常。

## 十一、开发顺序

建议按下面顺序推进：

1. 实现页范围解析工具
2. 实现 `split.ts`
3. 做 `split-pdf-card.tsx`
4. 新建页面路由
5. 接入导航和首页入口
6. 补 metadata
7. 验证下载流程

## 十二、验收标准

- 可以上传一个 PDF
- 可以识别总页数
- 可以输入合法页范围
- 可以生成多个拆分结果
- 可以逐个下载拆分后的 PDF
- 页面可以独立访问
- 错误输入有清晰提示

## 十三、结论

在当前“不做服务端”的前提下，`Split PDF` 是非常合适的下一步：

- 技术风险低
- 可复用现有栈
- 用户需求明确
- 和压缩、合并形成稳定工具组合

第一版策略应该坚持：

- 只做浏览器内
- 只做页范围拆分
- 先跑通主流程
- 不提前做重交互编辑器

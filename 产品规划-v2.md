## 2. 产品定位

### 当前定位

FileSmaller 当前应定位为：

> Browser-first PDF tools for making files smaller and easier to upload.

中文理解：

> 一个优先在浏览器本地处理的 PDF 工具站，核心价值是让 PDF 更小、更适合上传、提交和分享。

### 不要改成什么

短期不要改成：

- 大而全 PDF 平台
- Adobe / iLovePDF / SmallPDF 的全量复刻
- 泛 ChatPDF
- 泛 AI 文档平台
- 需要登录才能用的 SaaS

原因：

- 当前流量还在验证期
- 项目的核心资产是 SEO 页面和浏览器内工具闭环
- 过早平台化会拖慢迭代
- 大而全会让定位变散

### 未来定位

如果 SEO 和工具使用数据跑通，FileSmaller 可以升级为：

> A privacy-first PDF toolkit for upload limits, document submission, and lightweight file workflows.

重点仍然是“文件上传/提交/分享前处理”，不是无限扩 PDF 功能。

## 3. 战略选择

当前项目未来有三种可选路线：

1. 继续做目标大小压缩和上传限制工具
2. 扩成大而全 PDF 工具站
3. 转成 AI PDF 文档解析工具

本项目选择：

## 主线：继续做上传限制场景下的 PDF 工具站

理由：

- 当前已有代码、页面、SEO 和用户路径都围绕这个方向
- “压缩、拆分、合并、转图片、图片转 PDF”都服务于文件提交前处理
- 这个定位比泛 PDF 工具站更窄，更容易建立页面集群
- 后续可以自然加服务端强压和批量能力作为商业化入口

## 副线：用开源引擎补强压缩效果

当前最大短板不是页面数量，而是：

- 浏览器端压缩上限
- 扫描件、大图片 PDF 压缩不稳定
- 严格目标大小不一定能达到

所以接下来要优先调研和验证：

- `qpdf / qpdf-wasm`
- `pdfcpu`
- Ghostscript 效果对比

原则：

> 先接成熟引擎，少从 0 写压缩算法。

## 暂缓：AI PDF

Docling、PDF to Markdown、文档结构化可以作为后期方向，但不进入当前主线。

启动条件：

- 自然搜索已有稳定点击
- PDF 工具主流程数据稳定
- 有用户表现出对解析、提取、总结的需求

## 4. GitHub 项目利用策略

详细调研记录见：`docs/GitHub开源复用调研.md`。

一句话结论：

> 不 fork 大项目替换 FileSmaller；保留当前 Next.js + Cloudflare Workers 项目，用成熟开源引擎补强压缩能力。

### 4.1 已经在用，继续保留

| 项目 | GitHub 热度 | 当前用途 | 规划 |
|---|---:|---|---|
| `mozilla/pdf.js` / `pdfjs-dist` | 约 `53k` stars | 扫描件渲染、PDF 页面转图、后续预览 | 继续使用 |
| `Hopding/pdf-lib` | 约 `8k` stars | PDF 读取、重写、合并、拆分、删除、重排、图片生成 PDF | 继续作为浏览器端基础库 |
| `jszip` | 成熟 npm 依赖 | 批量结果打包下载 | 继续使用 |

### 4.2 下一步优先验证

| 项目 | GitHub 热度 | 目标用途 | 是否上项目 |
|---|---:|---|---|
| `qpdf/qpdf` | 约 `5k` stars | PDF 结构优化、线性化、对象流处理、压缩补强 | 上 P1 第一技术验证 |
| `pdfcpu/pdfcpu` | 约 `8.7k` stars | 服务端 PDF 压缩、拆分、优化 | 上 P1 第二技术验证 |
| `qpdf-wasm` | 非高 star 项目 | 浏览器端或 Worker 端 qpdf 能力 | 只做 spike，不直接押注 |
| `Ghostscript` / `ghostpdl` | 老牌项目，GitHub 为镜像 | 强力压缩扫描件和图片型 PDF | 只做效果对比，license 和部署确认前不上生产 |

### 4.3 只参考，不直接 fork

| 项目 | GitHub 热度 | 参考内容 | 不直接 fork 的原因 |
|---|---:|---|---|
| `Stirling-Tools/Stirling-PDF` | 约 `70k` stars | 功能地图、API、工具分类、强压/转换能力 | 太重，Java 自托管大平台，会覆盖当前轻工具站定位 |
| `alam00000/bentopdf` | 约 `5.5k` stars | 隐私优先表达、客户端处理体验 | AGPL/商业授权，且部分能力依赖 AGPL 组件，不适合直接拿代码 |
| `gotenberg/gotenberg` | 约 `10.6k` stars | 容器化文档转换 API、服务端架构 | 主能力是转换/生成 PDF，不是当前最急的压缩证据闭环 |
| `OmniTools` | 多工具站首页、导航、工具聚合方式 | PDF 压缩能力不是它的核心优势 |

### 4.4 明确不上或暂不上

| 方案 | 当前决策 | 原因 |
|---|---|---|
| 直接 fork Stirling-PDF 替换当前项目 | 不上 | 技术栈、部署模型和产品定位都不匹配 |
| 直接复制 BentoPDF / PDFCraft 代码 | 不上 | 授权和组件依赖风险高 |
| 直接把 Gotenberg 作为压缩核心 | 暂不上 | 它更适合文档转 PDF，不解决当前真实压缩效果验证 |
| 自研底层压缩算法 | 暂不上 | 违背“尽量不从 0 写”的原则，除非 qpdf / pdfcpu / Ghostscript 都不能满足 |

### 4.5 后期再考虑

| 项目 | 用途 | 启动条件 |
|---|---|---|
| `Docling` | PDF to Markdown / JSON / 结构化解析 | 进入 AI 文档阶段 |
| `pdfme` | 模板化 PDF 生成 | 做发票、证书、表单生成时再上 |

## 5.1 当前可做事项优先级

这里是基于当前产品功能、benchmark 结果和“尽量复用开源项目，不从 0 写”的执行优先级。

### P0 必须先做：补真实样本和画质结论

优先级最高，因为它决定后面是否值得接 qpdf / pdfcpu / Ghostscript。

- 收集 `8-20` 个非敏感真实 PDF 样本，放入 `test-fixtures/pdf-compression/real-samples/`
- 先跑 `npm run samples:intake-kit` 生成采样交付目录和 checklist
- 用 `npm run samples:import -- --list` 查看缺口
- 用 `npm run samples:privacy-check -- --source <local.pdf> --write-report` 做导入前预检
- 如果预检发现元数据风险，用 `npm run samples:sanitize -- --source <local.pdf> --output <sanitized.pdf>` 清理后再预检
- 用 `npm run samples:import -- --slot <slot.pdf> --source <local.pdf>` 导入到 manifest 约定文件名
- 如果已按 manifest 槽位重命名好一批 PDF，用 `npm run samples:import -- --source-dir <dir> --dry-run` 先预览，再去掉 `--dry-run` 批量导入
- 跑 `npm run benchmark:pipeline`
- 跑 `npm run benchmark:external-setup`，确认 qpdf / pdfcpu / Ghostscript 是否能被本机或自定义二进制路径调用
- 从 `/dev/compression-benchmark` 手动导出浏览器 scanned 路径结果
- 在 `/dev/compression-benchmark` 点击 `Save suite files`，优先直接保存 browser suite
- 导出的浏览器 suite 必须包含 `Fixture mix`，并能区分真实样本和合成样本
- 如果只有旧版 suite，先跑 `npm run benchmark:browser-suite-normalize` 统一格式
- 跑 `npm run benchmark:browser-suite-check`，确认浏览器 suite 是新版格式
- 填写 `quality-review-template.md` 的 `Readable?` 和 `Submit-safe?`
- 跑 `npm run benchmark:decision-gate`，只有 `PASS` 后才进入强压引擎生产接入决策
- 跑 `npm run benchmark:engine-recommendation`，确认是否已经能推荐第一批外部引擎
- 跑 `npm run benchmark:status`，统一查看真实样本、浏览器 suite、外部引擎和门禁状态

当前状态：工具链、导入命令、浏览器 suite 保存、决策门禁和状态汇总已齐；样本为 `0/8`，浏览器 suite 格式为 `PASS`，决策门禁为 `BLOCKED`，这是预期状态。

### P1 最高价值开发：强压引擎 POC

样本补齐后马上做，不建议先继续堆新工具。

推荐顺序：

1. `qpdf-wasm / qpdf`：先验证结构优化、线性化、对象流处理，对 clean / mixed PDF 是否有补强
2. `pdfcpu`：验证服务端压缩和优化效果，适合作为后端 POC 候选
3. Ghostscript：只做效果对比，重点看扫描件和图片型 PDF，license / 部署确认前不上生产

本机或 CI 不需要把二进制固定写死在代码里，可用 `QPDF_BIN`、`PDFCPU_BIN`、`GS_BIN` 指向已下载的可执行文件。

自动推荐规则：

- 必须有真实样本外部压缩成功结果，否则推荐状态保持 `BLOCKED`
- qpdf / qpdf-wasm 风险最低，真实样本效果接近时优先选 qpdf
- pdfcpu 作为服务端候选，只有效果或部署明显优于 qpdf 时上生产 POC
- Ghostscript 只参与强压效果对比；即使效果最好，也必须先过 license、部署和画质复核，默认不直接推荐为生产引擎

接入原则：

- 能直接复用成熟引擎就复用
- 只有这些引擎达不到目标时，才局部自研策略
- 不 fork Stirling-PDF / PDFCraft / BentoPDF 替换当前项目

### P1.5 已确认值得继续做：浏览器 Scanned PDF 路径

当前证据显示，浏览器 `Scanned PDF` 对图片型 / 扫描型 PDF 有实际价值。

已完成：

- 严格目标大小下触发 portal-limit scan 候选
- image-heavy 也可进入 portal-limit scan，不只限 scanned-heavy
- UI 已前置画质提示和 `Compare first page`
- 已新增 `npm run test:compression-strategy` 防止策略回退

下一步只需要靠真实样本验证清晰度，不需要重写这条路径。

### P2 转化优化：让用户更容易选对模式

在压缩证据继续补齐的同时可以并行做。

- 首页首屏继续聚焦 Compress PDF，不做泛工具首页
- 上传区默认说明更偏“upload limit / make PDF smaller”
- 结果区强化：
  - 原大小
  - 新大小
  - saved %
  - 是否达到 target size
  - 下一步推荐模式
- 未达到目标大小时，引导用户重试推荐模式，而不是只报失败
- GA4 验证上传、成功、失败、下载、重试、工具切换事件

### P3 暂缓：继续扩工具和 AI PDF

短期不优先做：

- PDF to Word
- OCR 平台
- ChatPDF
- 登录注册 / 会员系统
- 大而全 PDF editor

理由：当前最大瓶颈是核心压缩效果和真实样本证据，不是功能数量。

## 6. P1：压缩能力补强

P1 是当前最重要阶段。

目标：

- 让 FileSmaller 的核心压缩能力明显更强
- 特别是扫描件和图片型 PDF
- 尽量复用成熟引擎，不重写底层算法

建议路线：

### 6.1 保留当前浏览器端压缩

当前 `pdf-lib + pdfjs-dist` 路线有价值：

- 隐私好
- 成本低
- 速度快
- 适合普通文件

继续作为默认免费路径。

### 6.2 增加强压候选路径

优先验证顺序：

1. `qpdf-wasm`
2. `qpdf` 服务端
3. `pdfcpu`
4. Ghostscript

选择标准：

- 压缩效果
- 文件质量
- 处理速度
- Cloudflare Workers 兼容性
- 部署复杂度
- license 风险

### 6.3 产品表现

前端可以保持当前模式，但底层变成多引擎：

- Fast browser compression
- Strong compression
- Scanned PDF compression

文案仍然保守：

- 不承诺精确压到
- 不承诺无损
- 明确提示结果取决于文件内容

验收标准：

- 扫描件压缩效果比当前版本有明显提升
- 至少 3 类真实样本有量化改善
- 失败时能给出明确下一步建议

## 7. P2：工具流程和页面转化优化

## 11. 近期执行顺序

建议接下来按这个顺序：

1. 收集 8-20 个真实 PDF 样本，并用 `samples:import` 导入到 manifest 槽位
2. 跑 `npm run benchmark:pipeline` 和浏览器 `/dev/compression-benchmark`
3. 填写人工画质复核模板
4. 对比当前实现、qpdf、pdfcpu、Ghostscript 的压缩效果
5. 跑 `npm run benchmark:decision-gate`
6. 门禁通过后，选择一个强压引擎做 POC
7. 如果 POC 有明显提升，接入为 Strong / Server-side compression
8. 同步优化首页和核心工具页转化
9. 观察 Search Console 和 GA4 数据
10. 再决定是否扩新页面或新工具

一句话结论：

> `target-pdf-compress` 下一阶段不是重做，也不是换成别人的大项目，而是在现有 FileSmaller 基础上，用成熟开源引擎补强压缩效果，再用数据决定下一步扩张。

## 12. 发布策略

为了避免“可以发布”和“强压引擎已验证”混在一起，项目拆成两类发布门禁。

### 12.1 普通产品发布

可以发布的范围：

- 当前浏览器端 PDF 压缩
- `Scanned PDF` 浏览器 render-and-rebuild 路径
- 上传队列提示、目标大小提示、失败后的下一步建议
- SEO 页面、GA4 事件、开发 benchmark 工具
- 规划文档和开源复用调研

发布前命令：

```bash
npm run release:standard-check
```

只要 `release:readiness` 的 `Standard Release` 为 `PASS`，这类改动可以发。

### 12.2 强压引擎发布

不能直接发布的范围：

- 宣称 qpdf / pdfcpu / Ghostscript 已经作为生产强压引擎上线
- 宣称可以稳定达到严格目标大小
- 宣称服务端强压质量已验证

必须满足：

- 真实样本 `8/8`
- 浏览器 benchmark 有真实样本结果
- 人工画质复核没有 `TBD`
- 外部引擎有真实样本成功结果
- `npm run benchmark:decision-gate` 为 `PASS`
- `npm run benchmark:engine-recommendation` 给出推荐引擎

当前结论：

> 现阶段可以发布普通产品改进；不能把外部强压引擎作为生产能力发布或宣传。

发布执行清单见：`发布检查清单.md`。

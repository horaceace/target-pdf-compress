# P1-P2 技术方案

## 目标

今天的开发范围只覆盖两个阶段：

- `P1`：核心压缩能力升级
- `P2`：压缩流程体验升级

本阶段不做：

- 登录注册
- 支付
- 多工具矩阵
- OCR
- 服务端 API 平台化

当前目标很明确：

1. 让 PDF 压缩结果更强、更可感知
2. 让上传、压缩、下载这条主流程更顺

## 一、当前技术现状

当前项目技术基线：

- 前端框架：`Next.js`
- 语言：`TypeScript`
- 当前部署：`Cloudflare Workers`
- 当前压缩方式：浏览器内处理
- 当前核心依赖：`pdf-lib`

当前已具备：

- 单文件上传
- 基础压缩模式
- 压缩结果下载
- SEO 页面体系

当前主要问题：

- 压缩策略偏轻
- 对扫描件和图片型 PDF 不够强
- 没有批量压缩
- 缺少拖拽上传
- 缺少“再压一次”路径
- 缺少更完整的错误提示和结果展示

## 二、P1 范围定义

P1 只做“压缩本身”的升级。

### P1.1 压缩等级体系重构

目标：

- 让压缩模式更清晰
- 让用户理解不同模式的代价和收益

建议模式：

1. `light`
- 尽量保持质量
- 适合合同、简历、普通文档

2. `balanced`
- 体积和可读性平衡
- 作为默认模式

3. `strong`
- 明显缩小体积
- 适合上传、附件、门户

4. `extreme`
- 尽量压到更小
- 适合上传失败后的二次尝试

技术要求：

- 压缩模式不要只停留在文案层
- 每个模式对应真实参数差异

### P1.2 图片重采样策略

目标：

- 让图片型 PDF 真正有压缩差异

实现方向：

1. 识别 PDF 页内图像资源
2. 根据模式选择不同重采样策略
3. 在导出时使用不同图像质量参数

建议参数方向：

- `light`
  - 更高图片质量
  - 更少缩放
- `balanced`
  - 适度缩放
  - 适度质量压缩
- `strong`
  - 更明显缩放
  - 更低图像质量
- `extreme`
  - 更激进缩放
  - 允许可见质量损失

注意：

- `pdf-lib` 对“深度压缩扫描件”能力有限
- 当前阶段重点是“做得比现在强”，不是立刻达到 Ghostscript 级别

### P1.3 扫描件专项模式

目标：

- 单独优化图片密集型文件

产品表现上：

- 在 UI 中明确告诉用户：
  - `Scanned PDF mode`
  - 更适合证件、扫描表单、证明材料

技术策略：

- 对大图更激进缩放
- 优先压图，不强调视觉保真
- 在结果区显示：
  - `Best for image-heavy PDFs`

### P1.4 结果信息增强

压缩完成后至少展示：

- 原始文件大小
- 压缩后文件大小
- 压缩比例
- 节省体积
- 当前模式
- 是否建议“再压一次”

技术实现：

- 结果状态对象里增加字段
- 统一由前端结果面板渲染

## 三、P2 范围定义

P2 只做主流程体验，不扩产品边界。

### P2.1 拖拽上传

目标：

- 降低上传摩擦

实现方案：

1. 上传卡片支持 `dragenter`
2. 支持 `dragover`
3. 支持 `drop`
4. 拖拽时显示高亮态

要求：

- 保留原文件选择按钮
- 拖拽只是增强，不替代点击上传

### P2.2 批量压缩

目标：

- 从单文件工具升级为可工作工具

第一阶段建议：

- 支持一次上传多个 PDF
- 每个文件单独压缩
- 每个文件单独展示结果
- 支持全部下载

技术方案：

新增状态模型：

- `UploadJob`
  - `id`
  - `file`
  - `status`
  - `mode`
  - `originalSize`
  - `compressedSize`
  - `savedBytes`
  - `resultBlob`
  - `error`

状态流转：

1. `idle`
2. `queued`
3. `processing`
4. `success`
5. `error`

处理策略：

- 先串行处理，保证实现稳定
- 后续再考虑并发数量控制

### P2.3 上传限制与错误提示

需要覆盖的错误：

- 非 PDF 文件
- 空文件
- 文件解析失败
- 压缩失败
- 浏览器不支持

提示原则：

- 不要只显示技术错误
- 要给用户下一步建议

例如：

- `This PDF could not be processed. Try a smaller file or a different compression mode.`
- `Scanned PDFs may need stronger compression than the current browser-based version can provide.`

### P2.4 一键更强压缩

目标：

- 让用户在一次失败后不需要重新上传

实现方式：

- 结果区增加按钮：
  - `Try stronger compression`

逻辑：

- 如果当前模式不是 `extreme`
- 允许基于原始 `File` 再执行一次更强模式

### P2.5 结果区对比体验

结果区建议拆成两层：

1. 结果摘要
- before / after
- saved
- mode

2. 下一步动作
- download
- try stronger compression
- compress another file

如果是批量模式：

- 每个文件单独有结果卡片
- 顶部有总览信息

## 四、模块拆分建议

### 1. 压缩核心模块

建议文件：

- `lib/pdf/compress.ts`
- `lib/pdf/modes.ts`
- `lib/pdf/image-strategy.ts`

职责：

- 压缩模式定义
- 参数映射
- 执行压缩

### 2. 上传状态模块

建议文件：

- `lib/upload/types.ts`
- `lib/upload/jobs.ts`

职责：

- 批量任务模型
- 状态流转

### 3. UI 组件层

建议拆分：

- `components/upload-dropzone.tsx`
- `components/upload-card.tsx`
- `components/upload-job-list.tsx`
- `components/compression-result.tsx`

职责：

- 上传入口
- 拖拽区域
- 批量任务列表
- 结果展示

## 五、状态设计

### 单文件状态

适用于未进入批量模式前：

- `selectedFile`
- `mode`
- `status`
- `result`
- `error`

### 批量状态

建议直接过渡到统一的 `jobs[]` 模式：

- 单文件时 `jobs.length = 1`
- 多文件时 `jobs.length > 1`

这样能减少后续重构成本。

## 六、实现顺序

今天建议按这个顺序开发：

### Step 1

先重构压缩模式数据结构：

- 定义 `light / balanced / strong / extreme`
- 模式和 UI 文案解耦

### Step 2

改压缩核心：

- 补图片重采样策略入口
- 做扫描件偏强模式

### Step 3

重构上传状态为 `jobs[]`

- 先兼容单文件
- 为批量模式铺路

### Step 4

接入拖拽上传

### Step 5

补批量压缩 UI

### Step 6

补结果信息和一键再压一次

### Step 7

补错误提示和边界测试

## 七、风险与限制

### 1. 浏览器内压缩上限

这是当前方案的硬限制：

- 扫描件极限压缩效果有限
- 超大文件内存压力高
- 很难达到服务端强压效果

结论：

- 本阶段目标是显著优于当前版本
- 不是一次性解决所有扫描件问题

### 2. 批量模式的浏览器压力

如果一次上传很多大 PDF：

- 内存占用会上升
- 浏览器可能卡顿

所以第一版建议：

- 串行处理
- 限制批量数量
- 给出大文件提示

### 3. Cloudflare Workers 与本阶段关系不大

当前压缩主要在前端浏览器执行：

- 这轮改造主要影响前端代码
- 部署方式不需要大改

## 八、验收标准

P1 验收：

- 至少 4 个压缩模式可选
- 不同模式结果有明显差异
- 扫描件模式比当前版本更激进
- 结果区可展示压缩比例和节省体积

P2 验收：

- 支持拖拽上传
- 支持多文件批量压缩
- 支持逐个下载结果
- 支持一键更强压缩
- 错误提示清晰可理解

## 九、今天的建议结论

如果今天只能完成一部分，优先级是：

1. 压缩模式重构
2. 批量状态模型
3. 拖拽上传
4. 结果信息增强
5. 一键再压一次

最不该跳过的是：

- 模式真实差异化
- 批量状态设计

因为这两个决定后面代码是不是还能继续扩。
